manifest.json 설명
==================

이 파일은 UI5 애플리케이션의 설계도와 같습니다.
화면 코드가 실제 건물이라면, manifest.json은 그 건물이 어떤 이름을 가지며, 어떤 데이터와 연결되고, 어떤 화면 흐름을 따르는지 정해 두는 청사진이라고 이해하면 좋습니다.

JSON 형식은 표준적으로 주석을 허용하지 않습니다.
따라서 manifest.json 안에 직접 설명을 넣지 않고, 이 문서에서 각 영역의 의미를 차분히 정리합니다.


1. _version
-----------

"_version": "1.60.0"

manifest 파일 형식의 버전을 나타냅니다.
UI5 런타임은 이 값을 참고하여 manifest의 구조를 어떻게 해석할지 판단합니다.
일반 애플리케이션 개발에서는 자주 변경하지 않으며, 생성 도구가 정해 준 값을 유지하는 경우가 많습니다.


2. sap.app
----------

sap.app 영역은 애플리케이션의 신원과 데이터 소스를 설명합니다.
사람으로 비유하면 이름, 소개, 사용하는 언어, 연결된 외부 자료를 적어 둔 기본 인적 사항에 가깝습니다.

id 값인 "code.t0.ui5.review5"는 이 앱의 고유 네임스페이스입니다.
Component.js, View, Controller의 모듈 경로도 이 네임스페이스를 바탕으로 연결됩니다.

type은 "application"으로 지정되어 있습니다.
이는 이 프로젝트가 재사용 라이브러리가 아니라 실행 가능한 UI5 애플리케이션임을 뜻합니다.

i18n은 "i18n/i18n.properties"를 가리킵니다.
화면의 제목이나 안내 문구를 코드에 직접 박아 두지 않고, 언어 리소스 파일로 분리해 관리하겠다는 의미입니다.

title과 description은 "{{appTitle}}", "{{appDescription}}"처럼 중괄호 두 겹으로 감싸져 있습니다.
이는 i18n 리소스에서 해당 키를 찾아 실제 표시 문구로 치환한다는 뜻입니다.


3. dataSources
--------------

dataSources는 이 앱이 사용하는 외부 데이터의 위치와 성격을 정의합니다.
현재 두 가지 데이터 소스가 선언되어 있습니다.

ZUI_JHY_CARRIER_O2_VAN은 OData Annotation 소스입니다.
Annotation은 데이터의 의미, 화면 표현 힌트, 메타 정보 등을 보강하는 설명서라고 볼 수 있습니다.
localUri에는 로컬 개발 환경에서 참조할 annotation 파일 위치가 지정되어 있습니다.

mainService는 실제 업무 데이터를 제공하는 OData V2 서비스입니다.
uri는 "/sap/opu/odata/sap/ZUI_JHY_CARRIER_O2/"이며, 이 서비스에서 CarrierSet, ConnectionSet 같은 엔티티 집합을 읽어 옵니다.
settings의 annotations 항목은 mainService가 위의 annotation 소스와 함께 사용됨을 알려 줍니다.
localUri는 로컬 metadata.xml을 가리키며, 개발 도구가 서비스 구조를 이해하는 데 도움을 줍니다.


4. sap.ui
---------

sap.ui 영역은 UI 기술과 디바이스 지원 범위를 설명합니다.

technology가 "UI5"로 지정되어 있으므로, 이 앱은 SAPUI5/OpenUI5 기반 애플리케이션입니다.

icons 영역은 앱 아이콘, 즐겨찾기 아이콘, 모바일용 아이콘 등을 지정하는 자리입니다.
현재는 값이 비어 있으므로 별도 아이콘 리소스를 사용하지 않는 상태입니다.

deviceTypes에서는 desktop, tablet, phone이 모두 true입니다.
이는 이 앱이 데스크톱, 태블릿, 휴대폰 환경을 모두 지원 대상으로 삼고 있음을 의미합니다.
실제 화면에서는 FlexibleColumnLayout과 responsive 속성들이 이런 다양한 화면 크기에 대응하는 일을 돕습니다.


5. sap.ui5
----------

sap.ui5 영역은 UI5 애플리케이션 실행에 필요한 핵심 설정을 담습니다.
어떤 라이브러리를 쓰는지, 어떤 모델을 기본으로 둘지, 어떤 CSS를 읽을지, 라우팅을 어떻게 할지 모두 이곳에 모입니다.

flexEnabled가 true이면 UI5 Flexibility 기능을 사용할 수 있습니다.
이는 화면 변경이나 개인화 같은 확장 가능성을 염두에 둔 설정입니다.


6. dependencies
---------------

dependencies는 앱 실행에 필요한 UI5 버전과 라이브러리를 선언합니다.

minUI5Version은 "1.120.23"입니다.
이 앱이 최소한 해당 UI5 버전 이상에서 동작해야 함을 뜻합니다.

libs에는 sap.m, sap.ui.core, sap.f, sap.uxap, sap.ui.table이 포함되어 있습니다.
sap.m은 Button, Table, Page 같은 기본 모바일 친화 컨트롤을 제공합니다.
sap.f는 FlexibleColumnLayout과 DynamicPage 같은 Fiori 레이아웃을 제공합니다.
sap.uxap은 ObjectPageLayout처럼 상세 화면을 섹션으로 구성하는 컨트롤을 제공합니다.
sap.ui.core는 UI5의 근간이 되는 핵심 기능을 담당합니다.
sap.ui.table은 테이블 관련 기능을 위한 라이브러리입니다.


7. contentDensities
-------------------

compact와 cozy가 모두 true입니다.

compact는 데스크톱처럼 마우스로 조작하는 환경에 어울리는 촘촘한 밀도입니다.
cozy는 터치 환경에 적합하도록 조금 더 여유 있는 간격을 제공합니다.
둘 다 허용하면 실행 환경에 따라 UI5가 적절한 밀도를 선택할 수 있습니다.


8. models
---------

models 영역은 앱에서 사용할 모델들을 정의합니다.

i18n 모델은 ResourceModel입니다.
화면의 다국어 문구를 i18n 파일에서 읽어 오며, View에서는 {i18n>title} 같은 형태로 사용할 수 있습니다.

빈 문자열 ""로 선언된 모델은 기본 모델입니다.
이 모델은 mainService 데이터 소스에 연결되어 있습니다.
따라서 View에서 {Carrid}, {Carrname}, {to_Connection}처럼 모델 이름 없이 쓰는 바인딩은 이 OData 기본 모델을 기준으로 해석됩니다.

preload가 true이므로 앱 시작 시 모델 관련 리소스를 미리 준비하려는 의도를 가집니다.
초기 로딩 전략을 안정적으로 만드는 설정이라고 볼 수 있습니다.


9. resources
------------

resources의 css에는 "css/style.css"가 등록되어 있습니다.
이 파일은 앱 전체에 적용할 사용자 정의 스타일을 담는 자리입니다.
표준 UI5 테마만으로 부족한 세부 표현을 보완할 때 사용합니다.


10. routing
-----------

routing은 이 앱의 화면 이동 규칙을 정의합니다.
주소가 어떤 패턴일 때 어떤 View를 어느 컬럼에 보여 줄지 결정하는 매우 중요한 부분입니다.

config 영역의 controlId는 "idFlexibleColumnLayout"입니다.
이는 라우터가 App.view.xml 안의 FlexibleColumnLayout을 화면 배치의 중심 컨트롤로 사용한다는 뜻입니다.

controlAggregation은 기본값으로 "pages"가 지정되어 있지만, 각 target에서는 beginColumnPages, midColumnPages, endColumnPages를 따로 사용합니다.
이 설정 덕분에 Carrier는 첫 번째 컬럼, Connection은 두 번째 컬럼, Flight는 세 번째 컬럼에 들어갑니다.

path와 viewPath는 "code.t0.ui5.review5.view"입니다.
라우터가 View 이름을 찾을 때 이 경로를 기준으로 XML View를 불러옵니다.

async가 true이므로 View는 비동기 방식으로 로딩됩니다.
현대 UI5 애플리케이션에서는 초기 응답성과 유지보수성을 위해 비동기 로딩을 권장합니다.


11. routes
----------

routes는 URL 패턴과 target의 연결 규칙입니다.

RouteCarrier
pattern은 ":layout:"입니다.
layout 값이 있어도 되고 없어도 되는 선택적 경로처럼 동작합니다.
이 route는 TargetCarrier만 표시하므로 항공사 목록 중심의 시작 화면입니다.

RouteConnection
pattern은 "connection/{carrid}/{layout}"입니다.
carrid는 선택한 항공사의 키이며, layout은 FlexibleColumnLayout의 현재 배치를 나타냅니다.
target은 TargetCarrier와 TargetConnection 두 개입니다.
따라서 목록 컬럼과 항공사 상세 컬럼이 함께 표시됩니다.

RouteFlight
pattern은 "flight/{carrid}/{connid}/{layout}"입니다.
carrid와 connid를 통해 특정 연결편을 식별하고, layout으로 세 컬럼 배치를 제어합니다.
target은 TargetCarrier, TargetConnection, TargetFlight 세 개입니다.
따라서 항공사 목록, 항공사 상세, 항공편 상세가 함께 표시되는 구조가 됩니다.


12. targets
-----------

targets는 실제로 어떤 View를 어느 Aggregation에 넣을지 정의합니다.

TargetCarrier는 Carrier View를 beginColumnPages에 배치합니다.
이는 FlexibleColumnLayout의 첫 번째 컬럼입니다.

TargetConnection은 Connection View를 midColumnPages에 배치합니다.
이는 두 번째 컬럼이며, 선택된 항공사의 상세와 연결편 목록을 보여 줍니다.

TargetFlight는 Flight View를 endColumnPages에 배치합니다.
이는 세 번째 컬럼이며, 선택된 연결편의 날짜별 운항 정보와 다운로드 기능을 제공합니다.


13. rootView
------------

rootView는 앱이 처음 시작될 때 가장 먼저 생성되는 View입니다.

viewName은 "code.t0.ui5.review5.view.App"입니다.
즉, App.view.xml이 이 애플리케이션의 바깥 틀이 됩니다.

id는 "App"이며, type은 "XML"입니다.
async가 true이므로 루트 View도 비동기 방식으로 생성됩니다.

정리하면, 이 프로젝트의 manifest.json은 App.view.xml의 FlexibleColumnLayout을 중심으로 세 컬럼 화면을 구성하고,
Carrier, Connection, Flight 화면을 route와 target 규칙에 따라 적절한 컬럼에 배치하도록 설계되어 있습니다.
