## 1. 보고서 목적

이 문서는 `[MM] 견적 비교 및 거래선 채택` Fiori 프로그램의 1차 개발 완료 범위를 정리하고, 2차 개발 설계서 작성 시 이어서 검토해야 할 항목을 정리하기 위해 작성한다.

1차 개발의 핵심 목적은 다음이었다.

- RFQ Header를 조회한다.
- 선택 RFQ의 RFQ Item을 조회한다.
- RFQ Item별 MQ 후보를 비교한다.
- MQ별 금액 비교 차트를 제공한다.
- MQ 상세정보를 팝업으로 확인한다.
- RFQ Item당 MQ 1건을 거래선으로 채택한다.
- 이미 채택된 MQ를 채택취소한다.

PO 생성은 최초 설계서에서 2차 개발 범위로 분리되어 있었고, 현재 1차 개발에는 포함하지 않았다.

## 4. 현재 화면 구조

현재 화면의 최상위 레이아웃은 `sap.f.FlexibleColumnLayout`이다.

```text
Main.view.xml
└─ sap.f.FlexibleColumnLayout
   ├─ Begin Column
   │  └─ sap.uxap.ObjectPageLayout
   │     ├─ Header: 조회조건
   │     └─ Section: KPI + RFQ Header 목록
   └─ Mid Column
      └─ sap.uxap.ObjectPageLayout
         ├─ Header: 선택 RFQ 요약 정보
         ├─ Section 1: RFQ Item 목록
         ├─ Section 2: 금액 비교 차트
         └─ Section 3: MQ 비교 목록
```

처음에는 Mid 영역을 탭 구조로 구상했으나, 사용자 테스트 피드백을 반영해 `ObjectPageSection` 기반 세로 스크롤 구조로 변경했다. 사용자는 AnchorBar에서 `RFQ Item 목록`, `금액 비교 차트`, `MQ 비교 목록`을 선택해 해당 섹션으로 이동할 수 있다.

## 11. 2차 개발 전 Backend 선행 작업

2차 개발에서 PO 생성을 구현하려면 UI5보다 먼저 SEGW/OData 계약을 확정해야 한다.

현재 1차 OData는 채택/채택취소까지만 처리한다.

| 현재 EntitySet | 역할 |
|---|---|
| `QuotationItemSet` | MQ 채택/채택취소 |
| `MQCompareSet` | PO 생성 여부 표시만 가능 |
| `RFQItemSet` | PO 생성 여부와 Item 상태 표시만 가능 |

PO 생성은 단순 상태 변경이 아니라 새로운 구매문서 Header/Item을 생성하는 업무 처리다. 따라서 별도 OData 처리가 필요하다.

## 12. PO 생성 OData 설계 시 결정할 항목

2차 설계서 작성 전에 다음 항목을 먼저 확정하는 것이 좋다.

| 결정 항목 | 권장안 |
|---|---|
| OData 방식 | `POCreateSet` EntitySet의 `CREATE_ENTITY` 또는 Function Import |
| UI5 호출 방식 | `oModel.create("/POCreateSet", payload, ...)` 권장 |
| 생성 기준 | 채택된 MQ 기준 |
| 최소 입력 Key | `RfqNo`, `RfqItem`, `MqNo`, `MqItem` |
| 성공 응답 | 생성된 `PoNo`, `PoItem`, 메시지 |
| 실패 응답 | `/IWBEP/CX_MGW_BUSI_EXCEPTION`으로 업무 오류 반환 |
| 후속 재조회 | PO 생성 성공 후 Header → Item → MQ 재조회 |

개인적으로는 PO 생성은 `QuotationItemSet`의 `ActionType = "PO_CREATE"`처럼 억지로 확장하기보다, 별도 `POCreateSet`을 만드는 편이 더 명확하다. 이유는 채택/취소는 MQ Item의 상태 변경이고, PO 생성은 구매문서 생성이라는 업무 의미가 다르기 때문이다.

## 13. PO 생성 Backend 검증 규칙 후보

SEGW/DPC_EXT에서 PO 생성 로직을 만들 때 최소한 아래 검증이 필요하다.

| 검증 | 설명 |
|---|---|
| 채택 MQ 존재 | 해당 RFQ Item에 `SELIDC = X`인 MQ가 있어야 함 |
| 미응답 MQ 차단 | 단가 또는 견적수량이 0 이하이면 생성 불가 |
| 이미 PO 생성 차단 | 같은 MQ 또는 같은 RFQ Item에 이미 PO가 있으면 중복 생성 불가 |
| 삭제 데이터 차단 | Header/Item의 `LVORM` 확인 |
| 공급업체 확인 | MQ Header의 공급업체가 존재해야 함 |
| 자재/플랜트/저장위치 확인 | PO Item 생성에 필요한 조직/자재 정보 확인 |
| 수량/단가 확인 | PO 생성 가능한 수량, 단가, 가격단위 확인 |
| 후속 참조 저장 | PO Item에 원본 MQ 번호/품목을 `QTNFN/QTNFP`에 저장 |

## 14. PO 생성 OData 응답 필드 후보

2차 UI5에서 화면을 자연스럽게 갱신하려면 Backend 응답에 최소 아래 필드가 있으면 좋다.

| 필드 | 설명 |
|---|---|
| `RfqNo` | 원본 RFQ 번호 |
| `RfqItem` | 원본 RFQ Item |
| `MqNo` | 채택 MQ 번호 |
| `MqItem` | 채택 MQ Item |
| `PoNo` | 생성 PO 번호 |
| `PoItem` | 생성 PO Item |
| `MessageType` | S/E/W 등 |
| `MessageText` | 처리 결과 메시지 |

## 15. 2차 UI5 개발 후보

Backend OData가 준비된 뒤 UI5 2차 개발은 아래 순서가 적절하다.

| 순서 | 작업 | 설명 |
|---|---|---|
| 1 | PO 생성 버튼 위치 확정 | Footer 또는 RFQ Item/MQ 비교 영역 중 어디에 둘지 결정 |
| 2 | PO 생성 가능 조건 연결 | `SelectedRfqItem`, `SelectedMq`, `PoCreatedYn`, `CurrentAwardYn` 기준 |
| 3 | PO 생성 확인 팝업 | 생성 전 사용자 확인 |
| 4 | `POCreateSet` 호출 | Backend OData create 호출 |
| 5 | 성공 메시지 표시 | 생성 PO 번호 안내 |
| 6 | 성공 후 재조회 | Header, Item, MQ 비교 목록 재조회 |
| 7 | PO 생성 후 버튼 제어 | 채택/취소/PO 생성 버튼 비활성화 |
| 8 | PO 번호 표시 강화 | Header/Item/MQ Detail에서 PO 번호 확인 가능하게 정리 |

## 16. 유효성 검증과 MessagePopover 적용 시점

유효성 검증과 MessagePopover는 2차 PO 생성까지 끝낸 뒤 한 번에 적용하는 것이 좋다.

이유는 다음과 같다.

- 현재 1차 기능은 동작 흐름이 안정화되었다.
- 2차 PO 생성이 추가되면 검증 대상 버튼과 메시지가 늘어난다.
- MessagePopover는 조회조건 검증, 채택/취소 검증, PO 생성 검증을 한 번에 묶는 편이 중복이 적다.
- 기존 `delayed_po_monitor` 앱에서 이미 `MessagePopover`, `TableSelectDialog` 패턴을 성공적으로 사용했다.

## 19. 결론

1차 개발은 RFQ 기반 MQ 비교와 거래선 채택 업무를 수행할 수 있는 수준까지 완료되었다.  
2차 개발로 넘어가도 된다.

다만 PO 생성은 UI5만으로 진행할 수 없으므로, 먼저 SEGW에서 PO 생성을 위한 OData를 설계하고 구현해야 한다. Backend OData 계약이 확정된 뒤 UI5 2차 개발 설계서를 작성하는 순서가 가장 안전하다.

권장 순서는 다음과 같다.

```text
1. PO 생성 Backend OData 설계
2. SEGW / DPC_EXT 구현
3. Gateway Client로 PO 생성 테스트
4. 2차 UI5 설계서 작성
5. UI5 PO 생성 화면/로직 개발
6. 유효성 검증 + MessagePopover 적용
7. Search Help 적용
8. 최종 QA
```
