## 3. 현재 화면/데이터 구조 요약

최상위 화면은 `sap.f.FlexibleColumnLayout`이다.

```text
Main.view.xml
└─ sap.f.FlexibleColumnLayout
   ├─ Begin Column
   │  └─ RFQ Header 조회조건, KPI, RFQ Header 목록
   └─ Mid Column
      ├─ 선택 RFQ 요약 Header
      ├─ RFQ Item 목록
      ├─ 금액 비교 Chart
      └─ MQ 비교 목록
```

`view` 모델은 화면 제어 상태를 담당한다.

```js
{
  Busy: false,
  AdvancedFilterVisible: false,
  FclLayout: "OneColumn"
}
```

`work` 모델은 조회 결과와 선택 상태를 담당한다.

```js
{
  Kpi: {
    NotAwarded: 0,
    PartiallyAwarded: 0,
    Awarded: 0,
    PoCreated: 0
  },
  RfqHeaderCount: 0,
  RfqHeaders: [],
  RfqItems: [],
  MqCompareRows: [],
  ChartRows: [],
  MqDetail: {},
  SelectedRfq: {},
  SelectedRfqItem: {},
  SelectedMq: {}
}
```

## 5. 이번 미커밋 개발분 상세

### 5.1 채택 처리

`onSaveAward`가 현재 선택된 MQ를 기준으로 `QuotationItemSet`에 MERGE를 호출한다.

```text
onSaveAward
└─ SelectedMq 확인
   ├─ 없으면 msgSelectMq Toast
   └─ 있으면 MessageBox.confirm
      └─ OK 선택 시 _updateQuotationItem(MqNo, MqItem, "AWARD", msgAwardSuccess)
```

실제 OData 호출 구조는 다음과 같다.

```text
MERGE /sap/opu/odata/SAP/ZGWD3MM0005_SRV/QuotationItemSet(MqNo='...',MqItem='...')
Body:
{
  "MqNo": "...",
  "MqItem": "...",
  "ActionType": "AWARD"
}
```

설계서 기준으로 채택은 Function Import가 아니라 `QuotationItemSet` 단건 Update/MERGE다. Backend에서 `SELIDC = X` 반영과 기존 채택 MQ 정리를 처리한다.

### 5.2 채택 성공 후 재조회

채택 성공 후에는 화면 전체를 닫지 않고, 같은 RFQ Item 컨텍스트에서 아래 순서로 재조회한다.

```text
_refreshAfterAward
├─ 현재 SelectedRfq, SelectedRfqItem, FclLayout 저장
├─ _loadRfqHeaders({ keepComparisonContext: true })
├─ _loadRfqItemsForRfq(sRfqNo)
├─ 갱신된 RFQ Item 중 기존 RfqItem 재선택
├─ _loadMqCompareForRfqItem(...)
└─ 저장 전 FclLayout 복원
```

중요한 수정 사항은 `_loadRfqHeaders(mOptions)`의 `keepComparisonContext` 옵션이다.

일반 조회에서는 기존처럼 `_clearSelectionAndComparisonArea()`가 호출되어 `FclLayout = OneColumn`으로 초기화된다. 하지만 채택 성공 후 재조회에서는 이 초기화를 건너뛴다. 사용자는 데이터만 저장했을 뿐 화면을 닫은 것이 아니므로, Mid Column에 그대로 남는 것이 자연스럽다.

### 5.3 채택 후 레이아웃 유지 이슈

사용자 테스트에서 채택 자체는 DB `SELIDC = X`로 정상 반영되었지만, 화면이 Begin Column 레이아웃으로 돌아가는 문제가 있었다.

원인은 다음 흐름이었다.

```text
_updateQuotationItem
└─ _refreshAfterAward
   └─ _loadRfqHeaders
      └─ _clearSelectionAndComparisonArea
         └─ FclLayout = OneColumn
```

수정 후에는 채택 후 재조회에서만 Header 초기화를 보존형으로 실행한다.

```text
_refreshAfterAward
└─ _loadRfqHeaders({ keepComparisonContext: true })
   └─ Header 목록과 KPI만 갱신
```

추가로 재조회 완료 후 저장해둔 `FclLayout`을 다시 세팅한다. 따라서 일반 Mid 화면에서는 `TwoColumnsMidExpanded`, Mid 전체화면에서는 `MidColumnFullScreen` 상태를 유지한다.

## 11. 주의사항

- 설계서보다 현재 코드가 더 최신인 부분이 있다. 특히 MQ 상세 행별 버튼, 가격단위 컬럼, 자동추천, 채택 후 레이아웃 유지 흐름은 사용자 테스트 피드백을 반영한 내용이다.
- `_loadRfqHeaders()`를 수정할 때 일반 조회와 채택 후 재조회 흐름을 섞으면 안 된다. 일반 조회는 기존 선택을 초기화해야 하고, 채택 후 재조회는 기존 선택을 유지해야 한다.
- `SelectedMq`는 선택 가능한 MQ를 채택할 때만 사용한다. 이미 채택된 MQ 상세 조회는 행별 상세 버튼으로 가능하게 해두었다.
- 가격단위 `Peinh`는 Backend Metadata에 이미 존재한다고 판단되어 화면 컬럼만 추가했다. 별도 SEGW 수정은 하지 않았다.
- 현재 채택취소 버튼은 활성화 조건만 연결되어 있고, 실제 CANCEL 구현은 다음 단계 작업이다.
